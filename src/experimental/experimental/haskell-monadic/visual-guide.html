The monadic interface provides:

Flow Dynamics

The Unified Allocation Equation

Alloc(p, s, r) = min(qs · w(r)/Σw(e), |qr|)

Variables

p = Provider entity

s = Capacity slot (positive)

r = Recipient entity

qs = Capacity quantity

qr = Need quantity (negative)

Functions

w(r) = Weight function

Σw(e) = Sum of all weights

min() = Prevents over-allocation

📊 Allocation Process
1

Find Eligible Recipients

Apply filter Φ to find entities with matching sink potentials

2

Calculate Weights

Apply weight function w to each recipient (e.g., recognition × reputation)

3

Proportional Distribution

Distribute capacity proportional to weights: share = (weight / total_weights) × capacity

4

Apply Min Constraint

Cap allocation at recipient's need magnitude (prevents accumulation)

5

Record Immutably

Create AllocationRecord with timestamp, participants, and quantities

Key Properties

  • Proportional fairness — Allocation ratios equal weight ratios
  • No accumulation — Can't receive beyond need (min function)
  • Conservation — Total allocated ≤ source capacity
  • Auditability — Every flow recorded immutably

Types & Aggregation

⚡ Critical Distinction: Graph vs Potentials

Graph edges and potentials are SEPARATE data structures! They serve different purposes and are stored independently.

📊 Graph (G) - Structure

Purpose: Define relational context and type hierarchy


What it stores:

data Graph = Graph
  { vertices :: Set EntityId
  , edges    :: Map EntityId (Set EntityId)
  }

-- edges maps: entity → {types of entity}
-- Example:
edges = Map.fromList
  [ ("alice", Set.fromList ["programmer"])
  , ("programmer", Set.fromList ["worker"])
  ]

Meaning: "Alice IS-A Programmer" (membership/type relationship)

🌊 Potentials (P) - Properties

Purpose: Define flow gradients and resource states


What it stores:

type Potentials = Map EntityId [Potential]

data Potential = Potential
  { potentialType :: PotentialType
  , magnitude     :: SignedQuantity
  , unit          :: Maybe Text
  , ...
  }

-- Example:
potentials = Map.fromList
  [ ("alice", [Potential "meals" (-20) ...])
  , ("kitchen", [Potential "meals" 100 ...])
  ]

Meaning: "Alice HAS-NEED-FOR 20 meals" (flow potential)

🔍 Why They're Separate

The Complete Commons Structure: 

data Commons = Commons
  { graph      :: Graph           -- ← Edges: WHO is related to WHOM
  , potentials :: Potentials      -- ← Properties: WHAT each vertex has/needs
  , history    :: History         -- ← Records: WHAT actually flowed
  }

Think of it like a database with two tables:

Table 1: Relationships (Graph)

| Entity      | Types          |
|-------------|----------------|
| alice       | [programmer]   |
| bob         | [programmer]   |
| programmer  | [worker]       |
| worker      | [person]       |

Table 2: Properties (Potentials)

| Entity      | Type    | Magnitude |
|-------------|---------|-----------|
| alice       | meals   | -20       |
| bob         | meals   | -30       |
| kitchen     | meals   | +100      |
| alice       | skill   | +7.5      |

Key Points:

  • ✅ Edges define IS-A relationships (Alice is a Programmer)
  • ✅ Potentials define HAS/NEEDS properties (Alice needs 20 meals)
  • ✅ Same vertex can have multiple edges (many types) AND multiple potentials (many resources)
  • ✅ Edges are structural (graph topology), potentials are quantitative (signed magnitudes)
  • ✅ Aggregation functions use the graph to traverse and sum potentials along the path
💡 Concrete Example: How They Work Together
-- 1. GRAPH STRUCTURE (edges define relationships)
addVertexM "kitchen"
addVertexM "programmer"
addVertexM "alice"

addMembershipM "alice" "programmer"      -- Edge: alice → programmer
addMembershipM "programmer" "kitchen"    -- Edge: programmer → kitchen

-- Now we have:
-- Graph.edges = { "alice": {"programmer"}, "programmer": {"kitchen"} }

-- 2. POTENTIALS (properties on vertices)
addPotentialM "kitchen" $ Potential "meals" 100 ...     -- Kitchen HAS 100 meals
addPotentialM "alice" $ Potential "meals" (-20) ...     -- Alice NEEDS 20 meals

-- Now we have:
-- Potentials = { "kitchen": [Potential "meals" 100], "alice": [Potential "meals" -20] }

-- 3. ALLOCATION (uses BOTH)
performAllocationM "kitchen" "meals" noFilter sinkMagnitude "..." "..."

-- What happens:
-- a) Look up kitchen's POTENTIAL: finds +100 meals (from Potentials map)
-- b) Find recipients via GRAPH: members("kitchen") = {"programmer"} (from Graph.edges)
-- c) Recurse: members("programmer") = {"alice"} (from Graph.edges)
-- d) Check alice's POTENTIAL: finds -20 meals (from Potentials map)
-- e) Allocate: alice gets min(100 * (20/20), 20) = 20 meals

-- The GRAPH tells us WHO can receive (membership structure)
-- The POTENTIALS tell us HOW MUCH they need/have (flow magnitudes)

🎯 The Key Insight

Edges are not potentials. Edges define the topology of who's connected to whom. Potentials are attributes attached to vertices that describe flow gradients. The allocation algorithm traverses the graph structure (following edges) to find eligible recipients, then reads their potentials (from a separate map) to determine how much to allocate.


Think: Graph = skeleton, Potentials = flesh. The skeleton defines the shape and connections. The flesh defines the actual quantities flowing through those connections.

🔄 The Type/Instance Duality

In Commons, any vertex can be both a type AND an instance simultaneously. There's no separate type system—types are just vertices with incoming membership edges.


As a Type (has members)

-- Programmer has members
members(Programmer) = {Alice, Bob}

-- Check if it's a type
isType(Programmer) = True

As an Instance (has types)

-- Programmer is a member of Worker
types(Programmer) = {Worker}

-- Check if it's an instance
isMember(Programmer) = True

Key insight: Programmer can have its own potentials (e.g., average skill level) while also serving as a category for Alice and Bob!

Membership Relationships

🕸️ Graph Structure

An edge e₁ → e₂ means "e₁ is a member of e₂" or equivalently "e₂ is a type of e₁"


-- Example hierarchy
Alice → Programmer → Worker → Person
Bob → Programmer → Worker → Person

-- Graph operations
types(Alice) = {Programmer}
members(Programmer) = {Alice, Bob}
ancestors(Alice) = {Programmer, Worker, Person}
descendants(Person) = {Worker, Programmer, Alice, Bob}

Returns immediate types (parents) of entity e

types :: Graph -> EntityId -> Set EntityId

-- Example
types(g, "alice") = {"programmer"}

members(t)

Returns immediate members (children) of type t

members :: Graph -> EntityId -> Set EntityId

-- Example
members(g, "programmer") = {"alice", "bob"}

ancestors(e)

Transitive closure upward—all types reachable from e

ancestors :: Graph -> EntityId -> Set EntityId

-- Example

  {"programmer", "worker", "person"}

descendants(t)

Transitive closure downward—all members reachable from t

descendants :: Graph -> EntityId -> Set EntityId

-- Example

  {"programmer", "alice", "bob"}

Aggregation Mechanics

📊 Upward Aggregation (Agg_↑)

Aggregate potentials UP through category relationships

Agg_↑(v, τ) = Σ_{c ∈ Categories(v)} (Σ_{p ∈ P(c), type(p) = τ} p.q)

-- Haskell implementation
aggregateUp :: Graph -> Potentials -> EntityId -> PotentialType -> SignedQuantity
aggregateUp g pots eid ptype =
  let categoryIds = Set.toList (types g eid)
      magnitudes = [ magnitude p
                   | cid <- categoryIds
                   , p <- getPotentials pots cid
                   , potentialType p == ptype
                   ]
  in sum magnitudes

Example:

-- Alice is a member of Programmer
-- Programmer has skill_level potential of +7.5

aggregateUp(g, pots, "alice", "skill_level") = 7.5

-- Alice inherits the skill_level from Programmer!
📉 Downward Aggregation (Agg_↓)

Aggregate potentials DOWN through participant relationships

Agg_↓(c, τ) = Σ_{v ∈ Participants(c)} (Σ_{p ∈ P(v), type(p) = τ} p.q)

-- Haskell implementation
aggregateDown :: Graph -> Potentials -> EntityId -> PotentialType -> SignedQuantity
aggregateDown g pots cid ptype =
  let participantIds = Set.toList (members g cid)
      magnitudes = [ magnitude p
                   | eid <- participantIds
                   , p <- getPotentials pots eid
                   , potentialType p == ptype
                   ]
  in sum magnitudes

Example:

-- Programmer has members: Alice (-10 hours), Bob (-15 hours)

aggregateDown(g, pots, "programmer", "tutoring") = -25 hours

-- Total tutoring need for all Programmers!

aggregateSourcesDown

Sum only positive (source) potentials from members

aggregateSourcesDown g pots "kitchen" "meals"
-- Returns sum of all +meals from kitchen members

aggregateSinksDown

Sum only negative (sink) potentials from members (returns absolute value) 

aggregateSinksDown g pots "kitchen" "meals"
-- Returns |sum of all -meals| from kitchen members

Proportional Allocation Calculation

🧮 Step-by-Step Calculation
1

Find Eligible Recipients

eligibleRecipients :: Graph -> Potentials -> EntityId -> PotentialType -> Filter -> Set EntityId
eligibleRecipients g pots provider ptype filterFn =
  let participantIds = members g provider
      eligible = Set.filter
        (\eid ->
          let ctx = FilterContext g pots eid
          in filterFn ctx && hasMatchingSink pots eid ptype
        )
        participantIds
  in eligible

Get all members of provider, filter by constraints, check for matching sink potentials

2

Compute Weights

computeWeights :: Graph -> Potentials -> PotentialType -> Weight -> Set EntityId -> Map EntityId Double
computeWeights g pots ptype weightFn eligible =
  Map.fromList
    [ (eid, weight)
    | eid <- Set.toList eligible
    , let ctx = WeightContext g pots eid ptype
    , let weight = weightFn ctx
    , weight > 0
    ]

Apply weight function to each eligible recipient (e.g., w(r) = recognition × reputation)

3

Normalize to Shares

normalizeShares :: Map EntityId Double -> Map EntityId Double
normalizeShares weights
  | Map.null weights = Map.empty
  | otherwise =
      let total = sum (Map.elems weights)
      in if total > 0
         then Map.map (/ total) weights
         else Map.empty

Divide each weight by total to get proportional shares (sum = 1.0)

4

Allocate with Min Constraint

allocations = Map.mapWithKey
  (\eid share ->
    let sinkMag = case getPotential pots eid ptype of
          Just p  -> abs (magnitude p)
          Nothing -> 0
        proportional = source * share
    in min proportional sinkMag
  )
  shares

Multiply source by share, but cap at recipient's need magnitude

🎯 Key Insights

  • No special type system — Types are just vertices with members
  • Bidirectional relationships — types(e) and members(t) are inverses
  • Transitive aggregation — Potentials flow up and down the hierarchy
  • Proportional fairness — Allocation ratios exactly match weight ratios
  • Min prevents accumulation — Can't receive more than your sink magnitude
  • Everything composes — Build complex hierarchies from simple edges

The Monadic API

CommonsM Monad

newtype CommonsM a = CommonsM
  { unCommonsM :: StateT Commons (WriterT [AllocationRecord] IO) a
  }

A monad transformer stack combining:

  • StateT Commons — Thread Commons state
  • WriterT [AllocationRecord] — Accumulate history
  • IO — For timestamps and effects

State Operations

addVertexM :: EntityId -> CommonsM ()
addMembershipM :: EntityId -> EntityId -> CommonsM ()
addPotentialM :: EntityId -> Potential -> CommonsM ()

Modify the Commons state automatically

Allocation Operation

performAllocationM
  :: EntityId
  -> PotentialType
  -> Filter
  -> Weight
  -> Text
  -> Text
  -> CommonsM (Map EntityId SignedQuantity)

Allocate and auto-record to history

Query Operations

getGraph :: CommonsM Graph
getPotentials :: CommonsM Potentials
getHistory :: CommonsM History
getCommons :: CommonsM Commons

Access current state

Running the Monad

runCommonsM :: CommonsM a -> Commons -> IO (a, Commons, [AllocationRecord])
evalCommonsM :: CommonsM a -> Commons -> IO (a, Commons)
execCommonsM :: CommonsM a -> Commons -> IO Commons

Execute monadic computations

💡 Example: Monadic Workflow
example :: CommonsM ()
example = do
  -- Add entities
  addVertexM "kitchen"
  addVertexM "alice"
  
  -- Add relationships
  addMembershipM "alice" "kitchen"
  
  -- Add potentials
  addPotentialM "kitchen" $ Potential "meals" 100 Nothing Nothing Map.empty
  addPotentialM "alice" $ Potential "meals" (-20) Nothing Nothing Map.empty
  
  -- Perform allocation (automatically recorded!)
  allocations <- performAllocationM
    "kitchen"
    "meals"
    noFilter
    sinkMagnitude
    "no filter"
    "proportional"
  
  -- Query results
  history <- getHistory
  liftIO $ print (length history)  -- Shows 1 record

The Global Commons

🌍 From Local Allocation to Global Coordination

This isn't just a resource allocation algorithm. It's a mathematical foundation for organizing human cooperation at any scale—from a family kitchen to a planetary economy. By making fairness, transparency, and sustainability mathematically inevitable rather than socially enforced, we create a commons that naturally resists capture, accumulation, and exploitation.

🏛️ What Makes This a "Commons"?

🔓

Open Access

Anyone can join the graph by creating a vertex and declaring edges. There's no gatekeeper deciding who gets to participate.


Traditional: Membership requires approval, credentials, or payment.

Commons: Existence in the graph IS membership. You participate by being.

⚖️

Proportional Fairness

Allocation ratios are determined by mathematical formulas, not by power, wealth, or social status.


Traditional: Those with more resources get more access.

Commons: Your share equals your weight divided by total weights—pure math.

🚫

No Accumulation

The min function prevents receiving beyond need. You can't hoard resources for later or accumulate beyond your sink magnitude.


Traditional: Accumulation is encouraged (savings, investment, wealth).

Commons: Accumulation is mathematically impossible—flows seek equilibrium.

📜

Complete Transparency

Every allocation is recorded immutably with full metadata. The entire history is auditable by anyone.


Traditional: Transactions are private or require special access to audit.

Commons: All flows are public record—radical transparency by default.

🔄

Dynamic Equilibrium

Potentials change over time as needs are satisfied and capacities regenerate. The system continuously seeks balance.


Traditional: Static ownership—once you have it, you keep it.

Commons: Flowing gradients—potentials update as reality changes.

🌐

Scale Invariance

The same formalism works for 2 people sharing a meal or 2 billion people coordinating planetary resources.


Traditional: Different systems for different scales (family, market, state).

Commons: One unified mathematics from micro to macro.

🌱 Emergent Properties

When you implement this formalism at scale, certain behaviors emerge naturally from the mathematics without being explicitly programmed:

1

Organic Reciprocity

If weight functions incorporate past contributions (from history records), people who provide when they have capacity naturally receive more when they have need. This creates gift economy dynamics without explicit debt tracking.

2

Reputation as Network Position

Recognition edges create a graph where reputation emerges from who trusts you, weighted by their own reputation. This is like PageRank for social capital—you can't buy it, only earn it through actual flows.

3

Self-Organizing Hierarchies

Type hierarchies form naturally as people create membership edges. Programmers group under Workers, Workers under People. These aren't imposed—they emerge from participation patterns.

4

Anti-Gaming Properties

You can't fake need (your sink potential must match actual consumption over time), can't accumulate beyond capacity (min function), and can't manipulate reputation (it's computed from the graph). Gaming the system requires gaming reality itself.

5

Automatic Resource Discovery

When you declare a need, the system automatically finds all sources with matching capacity through graph traversal. You don't search for providers—the graph connects you.

🔮 The Paradigm Shift

From Ownership to Stewardship

You don't "own" resources—you're a temporary steward of flow potentials. When your need is satisfied, the potential dissipates and flows elsewhere.

From Scarcity to Sufficiency

The system doesn't maximize individual accumulation—it optimizes for collective sufficiency. Everyone gets enough before anyone gets excess.

From Competition to Coordination

You don't compete for resources—you declare your needs and capacities, and the graph coordinates flows. Your gain doesn't require someone else's loss.

From Enforcement to Emergence

Fairness isn't enforced by rules or police—it emerges from the mathematics. You can't violate proportional fairness any more than you can violate conservation of energy.

From Centralization to Distribution

No central authority decides allocations—the graph structure itself IS the authority. Power is distributed through edges, not concentrated in institutions.

From Opacity to Transparency

Every flow is recorded immutably and publicly. You can't hide accumulation, manipulation, or unfair advantage—the history reveals all.
